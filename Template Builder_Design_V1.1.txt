1) Align with existing eivs.datasources
Existing table:
CREATE TABLE IF NOT EXISTS eivs.datasources (
  datasource_id      SERIAL PRIMARY KEY,
  name               TEXT NOT NULL UNIQUE,  -- e.g. CRM_DB, LOAN_CORE_DB
  datasource_type    TEXT NOT NULL,         -- &#39;postgres&#39;, &#39;snowflake&#39;, &#39;api&#39;, etc.
  connection_key     TEXT NOT NULL,         -- what Adapter/middleware uses to route
  description        TEXT,
  semantic_model_yaml TEXT,
  is_active          BOOLEAN NOT NULL DEFAULT TRUE
);

Decision

We do not create a separate template_builder.datasources.

All template/placeholder references to DS will use:

datasource_id INT REFERENCES eivs.datasources(datasource_id)


This keeps all connection routing and semantic model information in one place and lets Template Builder rely on existing middleware/Adapter.


Impact on our design

placeholders_registry.datasource_id becomes INT referencing eivs.datasources.
Any “AI prompt → SQL → execute” pipeline will use eivs.datasources.connection_key to call the appropriate backend/middleware.

No duplication, no drift.

2) Relation between intents and templates
You want: “for an intent, these are the templates I would be using to fulfil the needs of the intent.”
This calls for a many‑to‑many relationship:

One intent can be served by multiple templates (e.g., short summary vs detailed letter).
One template can be reused for multiple intents (e.g., generic transaction summary).

Conceptual schema
Assuming you already have an eivs.intents table (or similar) in the NLP codebase:
-- Linking table between intents and templates
CREATE TABLE eivs.intent_templates (
  intent_id    INT NOT NULL REFERENCES eivs.intents(intent_id),
  template_id  UUID NOT NULL REFERENCES template_builder.templates(template_id),
  is_primary   BOOLEAN NOT NULL DEFAULT FALSE,
  rank         INT NOT NULL DEFAULT 1,  -- ranking within an intent
  PRIMARY KEY (intent_id, template_id)
);

Runtime usage

NLP/middleware side:
The intent classifier resolves an intent_id (and maybe a confidence, entities, etc.).


Template Builder side:
Given intent_id and context (user type, channel, language), we pick:
The is_primary template, or
A template with the highest rank that matches output format and locale.





API

GET /v1/intents/{intentId}/templates
Returns templates mapped to that intent with rank and metadata.


For n8n and other consumers, you might not even need this API if they already know the template ID; but it’s very helpful inside the middleware itself.

You want: from an n8n workflow, call a REST API to generate a doc using a template and get it back.
Design: minimal, n8n‑friendly API

5) AI prompts as placeholders (core pattern)
You asked: “Can AI Prompts be a placeholder like a field, we can mention the data source for the prompt, based on the prompt we generate the SQL, get the data and populate the placeholder?”
Design: yes – AI Prompt Placeholder Type
We make AI‑driven placeholders a first‑class type:

In placeholders_registry:

generation_mode:

&#39;manual_sql&#39; → we store SQL directly.
&#39;llm_prompt&#39; → we store a natural language prompt.
(Optional) &#39;semantic_model&#39; → use semantic model definitions.


datasource_id INT REFERENCES eivs.datasources(datasource_id)

prompt TEXT – natural language description of what to fetch.

value_type, cardinality, classification, format_json, etc.




Example placeholder definition:
{
  &quot;name&quot;: &quot;loan_details&quot;,
  &quot;generation_mode&quot;: &quot;llm_prompt&quot;,
  &quot;datasource_id&quot;: 12,  // eivs.datasources entry
  &quot;prompt&quot;: &quot;Given loan_number, fetch principal, interest_rate, maturity_date from the core loan table&quot;,
  &quot;value_type&quot;: &quot;json&quot;,
  &quot;cardinality&quot;: &quot;table&quot;,
  &quot;classification&quot;: &quot;internal&quot;
}

Runtime pipeline
At render time, for each placeholder with generation_mode = &#39;llm_prompt&#39;:

Resolver sees loan_details in the template.
Resolver sends {prompt, runtime_params, datasource connection_key} to Adapter / SQL generation service.
Adapter:
Uses LLM + semantic_model_yaml to generate SQL.
Runs SQL Safety Gate to validate.
Executes against the datasource.


Adapter returns structured rows.
Resolver formats them according to value_type, cardinality, format_json.
Renderer uses these values to populate the template (scalar, list, table, etc.).

This integrates perfectly with your existing middleware — Template Builder never talks to DBs directly.

4. Intent‑driven + AI‑native flow (how it all connects)
Putting it together:

User query → intent (NLP):

Your NLP/middleware resolves intent_id and key entities (loan_number, customer_id, etc.).


Intent → template:

Middleware looks up intent_templates to pick the best template for the intent (primary, rank, matching locale/output).


Template → placeholders:

It loads template_builder.templates and corresponding template_placeholders → placeholders_registry to know which AI/manual fields to populate.


AI prompts → SQL → data:

For each placeholder with generation_mode = &#39;llm_prompt&#39;, call the Adapter/AI layer to generate SQL, validate via SQL Safety Gate, execute using eivs.datasources.connection_key, and return structured data.


Resolver builds context:

Group by datasource, batch, format values, and prepare context.values and context.datasets for the renderer.


Renderer generates document:

Use your pluggable renderers (HTML, DOCX, PDF via HTML, XLSX, Markdown) to produce the final artifact.


Middleware/n8n receives output:

Via /v1/documents/generate response (base64 or URL) and can send it via email, WhatsApp, portal, etc.



This gives you the “world‑class” intent‑driven + AI‑native behavior end‑to‑end.

3) Answer to your doubt: “Based on the datasource will the user get the tables and field to choose from?”
Yes – and this should be a first‑class UX feature.
Given your existing eivs.datasources structure (with semantic_model_yaml), there are two complementary ways to give the user table/field choices:
3.1. Use semantic models (preferred)

eivs.datasources.semantic_model_yaml can describe:
Business entities (e.g., Loan, Customer, Policy, Claim).
Their underlying tables/views.
Fields/attributes and their types.



Backend:

Expose an internal (or external if you want) API like:

GET /v1/datasources/{datasourceId}/schema

Response example:
{
  &quot;datasource_id&quot;: 12,
  &quot;entities&quot;: [
    {
      &quot;name&quot;: &quot;Loan&quot;,
      &quot;table&quot;: &quot;core.loans&quot;,
      &quot;fields&quot;: [
        { &quot;name&quot;: &quot;loan_number&quot;, &quot;type&quot;: &quot;string&quot;, &quot;label&quot;: &quot;Loan Number&quot; },
        { &quot;name&quot;: &quot;principal&quot;, &quot;type&quot;: &quot;number&quot;, &quot;label&quot;: &quot;Principal Amount&quot; },
        { &quot;name&quot;: &quot;interest_rate&quot;, &quot;type&quot;: &quot;number&quot;, &quot;label&quot;: &quot;Interest Rate&quot; }
      ]
    }
  ]
}


This response can be derived from semantic_model_yaml or from Adapter’s existing metadata endpoints.


Frontend behavior:

When user defines a placeholder or table:
They choose a datasource from a dropdown.
Builder calls /v1/datasources/{id}/schema.
UI shows a list/tree of entities/tables and fields with types and labels.
User can:
Click a field to create a scalar placeholder (loan_number, principal).
Select multiple fields to create a dataset placeholder or auto‑configured table.




For AI prompts, this schema is also provided to the LLM so it can generate correct SQL with field and table names.

This is very consistent with your original design, which already envisioned a multi‑datasource abstraction layer and logical models over physical schemas.  
3.2. Direct DB introspection (fallback / supplement)
If semantic models are not available:

For relational datasources (Postgres, Snowflake, etc.), you can:

Either call Adapter to expose system catalog (tables, columns, types) in a secure, filtered way.
Or maintain a pre‑built metadata cache in your middleware.



The UI experience is the same, but behind the scenes you are basing the list on actual DB catalog rather than semantic YAML.
3.3. How this plugs into placeholders and tables

While defining a placeholder:

User chooses datasource → sees entities/fields → picks field(s) → the builder:
Fills in datasource_id.
Optionally pre‑fills a template SQL or prompt.
Sets value_type and cardinality automatically based on field selection.




While designing a table block:

User chooses a dataset placeholder (with cardinality table).
Or user selects multiple fields first; builder can:
Suggest creating a dataset placeholder and table block together.





This gives a very powerful but intuitive experience: the user feels like they’re designing at the “Loan / Customer / Policy” level, while the engine uses semantic models and AI to generate safe, efficient queries under the hood.
