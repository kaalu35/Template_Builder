Tables and repeats

AI configures table columns and bindings automatically (or the user picks fields).


Layout design  

User can type:  
“Create a 2‑page loan closure summary with a big title, a customer details section, and a table of repayments, in English and French.”

Text, style, and multilingual support

Make the tone more formal/legal/marketing.

4.2 AI &amp; safety layer
AI works inside your guardrails, not outside:

For placeholders in generationmode = &#39;llmprompt&#39;, the placeholder holds the prompt and datasource; the engine uses the Adapter/LLM to generate SQL, passes it through Safety Gate, then executes.–  
For doc→template, layout, table, text, anomaly suggestions, AI writes into aisuggestions and uploadeddocuments; the core engine remains deterministic and governed.–  
All AI actions are logged with metadata for audits and analytics.–


4.3 Integration &amp; automation: plug‑and‑play
We design the integration layer to be boringly simple and very stable:

Optionally async with render_jobs and job status endpoints.––


Template &amp; registry APIs (for config/ops and partners):  

CRUD for templates, placeholders, marketplace items (for devs and admin tools).–


Webhooks  

Fire on key events: template published, job complete, anomaly detected, etc.


SDKs / connectors  

Lightweight client libraries (JS, Python, Java) wrapping /v1/documents/generate and common flows.–  
Example connectors/recipes for n8n, Make.com, Zapier, etc., using their HTTP + webhook primitives.–



Because the external surface is small and predictable, any app that “uses template tools out there” can integrate with you with almost no effort (or via prebuilt connectors), making your system a drop‑in upgrade over legacy mail‑merge engines.–––

4. Adapter service: how the backend gets real data
Purpose: The Template Builder backend never hits databases directly; it calls Adapter, which uses eivs.datasources and semantic models to run queries.
Key behaviors in Phase 1:

Endpoint to execute a query:

Input:
datasource_id
sql (for now, coming from semantic model / admin, not from business UI)
params (runtime parameters, e.g., customer_id)


Output:
rows (list of row objects).




Internals:

Look up datasource in eivs.datasources and use connectionkey to connect.
For Phase 1, treat SQL as trusted (from semantic model / admin), but structure the code so later:
AI‑generated SQL for llmprompt placeholders runs through your SQL Safety Gate before execution.


Log all queries for observability and debugging.



You can extend this service later to accept a logical request (entity + fields + filters), and have it generate SQL from semantic models, but Phase 1 can simply store SQL along with semantic metadata, keeping that complexity behind admin tools and the Adapter.

POST /ui/templates
GET /ui/templates/{id}
